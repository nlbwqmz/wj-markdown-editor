# 统一预览资源右键菜单体系设计

## 1. 背景

当前项目的预览资源右键能力已经出现两条分叉链路：

- 编辑页预览区已经有独立的资源右键菜单，只支持“在文件夹中打开 / 删除”两项，菜单展示由 `PreviewAssetContextMenu.vue` 负责，业务执行由 `EditorView.vue` 负责。
- 纯预览页虽然也能响应资源右键，但当前行为是直接执行“在文件夹中打开”，没有统一菜单层，也没有和编辑页共享右键动作定义。

这导致同样是预览资源右键，不同宿主页面的交互模型不一致，后续如果继续追加右键能力，逻辑会进一步分散到多个页面与组件中。

## 2. 本次目标

本次仅重构“预览域中的资源右键”逻辑，目标如下：

- 统一编辑页预览与纯预览页的资源右键架构。
- 两种预览宿主都改为“右键弹出菜单，再由菜单触发动作”。
- 保留当前已有业务动作，不新增任何用户可见的新能力。
- 保留编辑页预览与纯预览页的权限差异。

## 3. 非目标

本次明确不包含以下内容：

- 不新增标题、代码块、表格、文本选区等新的右键命中对象。
- 不新增复制路径、复制 Markdown 引用、复制 PNG/JPG、复制图片到剪贴板等新动作。
- 不改造 Electron 主进程已有的资源打开、资源删除、资源信息查询能力。
- 不改变编辑页中现有删除确认、多引用删除、会话上下文校验等业务语义。
- 不把本次体系上升为全应用右键平台，本次只覆盖预览域中的资源右键。

## 4. 用户可见行为

本次重构完成后，用户可见行为应收敛为：

- 编辑页预览区右键本地资源：
  弹出菜单，菜单项与当前行为保持一致，只包含“在文件夹中打开 / 删除”。
- 纯预览页右键本地资源：
  也弹出菜单，但只包含“在文件夹中打开”。
- 非资源区域：
  保持现状，不新增右键菜单。

## 5. 设计原则

- 上下文统一：同一种预览资源右键，先统一解析命中上下文，再决定宿主页面允许哪些动作。
- 动作收口：动作定义独立于菜单展示与具体页面逻辑，避免在组件模板里写大量页面判断。
- 宿主裁剪：编辑页预览与纯预览页的差异由宿主 profile 决定，而不是由菜单组件硬编码。
- 展示纯化：菜单组件只负责渲染菜单项与派发 action key，不负责业务执行。
- 渐进扩展：本次只支持资源上下文，但接口要允许后续继续接入代码块、标题等其他预览右键类型。

## 6. 方案对比

### 方案 A：继续在各页面分别维护资源右键

优点：

- 改动最小。

缺点：

- 编辑页与纯预览页逻辑继续分叉。
- 后续加入新右键对象时，会出现更多重复实现。
- 无法形成统一的动作模型与宿主裁剪机制。

结论：

- 不采用。

### 方案 B：直接做全应用统一右键平台

优点：

- 理论上最统一。

缺点：

- 当前只有预览资源右键场景足够成熟，其他区域没有足够现成逻辑可收口。
- 会引入超出本次需求范围的抽象，增加实现与回归成本。

结论：

- 当前阶段不采用。

### 方案 C：先做预览域统一资源右键体系

优点：

- 精准覆盖当前已有分叉场景。
- 能复用编辑页已有业务逻辑。
- 为后续扩展新的预览右键对象保留统一入口。

缺点：

- 需要先抽出一层上下文、动作与宿主策略。

结论：

- 采用本方案。

## 7. 架构设计

### 7.1 统一右键上下文

`MarkdownPreview.vue` 当前已经可以从 DOM 中解析资源右键信息，包含：

- `kind`
- `rawSrc`
- `rawPath`
- `resourceUrl`
- `occurrence`
- `lineStart`
- `lineEnd`
- `clientX`
- `clientY`

本次不改变资源元数据本身，只把右键事件统一包装成“预览上下文”对象。第一期上下文只支持资源：

```js
{
  type: 'resource',
  asset: {
    kind,
    rawSrc,
    rawPath,
    resourceUrl,
    occurrence,
    lineStart,
    lineEnd,
  },
  menuPosition: {
    x,
    y,
  },
}
```

除命中对象本身外，host 在菜单打开时还必须冻结一份 `actionContext`，至少包含：

- `sessionId`
- `documentPath`
- 当前上下文版本号或等价失效标记

其目的不是新增用户能力，而是保证“菜单打开后，如果文档快照已经切换，旧菜单上的动作不能再对新文档执行”。该约束对编辑页与纯预览页都成立，纯预览页不能因为只保留 `open-in-folder` 就跳过这层保护。

事件命名层面，本次建议直接从资源专用事件升级为更通用的预览上下文事件，由 `MarkdownPreview` 对外抛出统一的预览右键事件，避免后续接入标题、代码块等其他上下文时再次重命名事件。

### 7.2 统一动作定义

本次动作定义只收口已有动作：

- `resource.open-in-folder`
- `resource.delete`

动作定义层负责说明“这个上下文有哪些候选动作”，但不负责判断当前页面是否允许显示，也不负责实际执行。

### 7.3 宿主 Profile

本次定义两个宿主 profile：

- `editor-preview`
- `standalone-preview`

二者能力差异如下：

| Profile | 允许动作 |
|------|------|
| `editor-preview` | `resource.open-in-folder`、`resource.delete` |
| `standalone-preview` | `resource.open-in-folder` |

这样可以保证：

- 编辑页预览保留删除等编辑性动作。
- 纯预览页不出现任何会修改 Markdown 或删除本地资源的动作。

### 7.4 菜单展示层

现有 `PreviewAssetContextMenu.vue` 继续作为菜单展示组件，但职责要调整为纯展示：

- 输入：`open`、`x`、`y`、`items`
- 输出：`select(actionKey)`、`close()`

它不再直接暴露 `openExplorer`、`delete` 这类业务事件，而是只抛出统一 action key。

本次 `items` 模型的最小字段约定如下：

- `key`
- `label`
- `danger`

其中：

- `key` 用于回传统一 action key
- `label` 用于渲染菜单文案
- `danger` 用于删除类动作的危险样式标识

### 7.5 宿主执行层

最终动作由宿主页面执行：

- 编辑页宿主继续复用现有资源打开与删除状态机，并沿用当前的 actionContext 冻结、失效判断与 requestContext 生成方式。
- 纯预览页宿主只处理 `resource.open-in-folder`，但同样必须在菜单打开时冻结 actionContext，并在菜单点击时基于该冻结上下文生成 requestContext，而不是点击时再直接读取最新 store 快照。

这样可以保留现有复杂业务逻辑的成熟实现，又避免把页面差异写进公共菜单组件。

## 8. 数据流

统一数据流如下：

1. `MarkdownPreview` 捕获右键事件。
2. 解析命中资源，生成统一上下文对象。
3. host 页面根据自身 profile 计算最终菜单项。
4. 菜单组件渲染菜单项。
5. 用户点击菜单项后，host 页面根据 action key 调用现有业务逻辑。

### 编辑页预览

流程为：

1. 解析资源上下文。
2. 菜单打开时冻结 actionContext。
3. 使用 `editor-preview` profile。
4. 生成 `open-in-folder / delete` 菜单项。
5. 用户点击后，继续复用当前 `EditorView.vue` 中已有的打开目录、删除确认、多引用删除、会话校验逻辑。

### 纯预览页

流程为：

1. 解析资源上下文。
2. 菜单打开时冻结 actionContext。
3. 使用 `standalone-preview` profile。
4. 生成仅包含 `open-in-folder` 的菜单项。
5. 用户点击后，使用冻结的 actionContext 生成 requestContext，再调用资源打开目录逻辑。

## 9. 文件影响范围

本次预计涉及的核心文件如下：

- `wj-markdown-editor-web/src/components/editor/MarkdownPreview.vue`
  负责统一预览资源右键上下文的抛出。
- `wj-markdown-editor-web/src/components/editor/PreviewAssetContextMenu.vue`
  从资源专用菜单组件演进为接收通用菜单项模型的展示层。
- `wj-markdown-editor-web/src/views/EditorView.vue`
  作为 `editor-preview` host，继续承载现有资源删除与打开逻辑。
- `wj-markdown-editor-web/src/views/PreviewView.vue`
  作为 `standalone-preview` host，接入统一右键菜单，但仅开放只读动作。
- 新增预览右键相关 util
  负责统一上下文动作与 profile 裁剪逻辑。

## 10. 迁移顺序

建议按以下顺序落地，降低回归风险：

1. 抽取“资源上下文 -> 候选动作 -> profile 裁剪”的纯函数层。
2. 先让编辑页接入新动作层，保证现有行为不变。
3. 再让纯预览页改为“右键弹菜单”，菜单项只保留“在文件夹中打开”。
4. 最后把菜单组件职责收口为纯展示层，统一 action key 触发方式。

## 11. 错误处理与边界

- 资源上下文解析失败时，不弹出自定义菜单，其余交给浏览器或宿主默认 `contextmenu` 行为。
- 纯预览页即使拿到资源上下文，也不能出现删除动作。
- 编辑页中的删除动作仍然必须保留当前已有的会话上下文校验，避免切换文档后对旧资源继续执行删除。
- 纯预览页中的 `open-in-folder` 也必须基于菜单打开时冻结的 actionContext 执行，避免“菜单对着旧文档打开，点击时却按新文档路径解析相对资源”。
- 主进程中已有的 `stale-document-context`、`not-found`、`relative-resource-without-document` 等结构化裁决保持不变。

## 12. 测试要求

本次重构完成后，至少需要覆盖以下测试点：

- 编辑页预览右键资源时，仍然展示当前两项菜单。
- 纯预览页右键资源时，改为展示只读菜单，而不是直接执行打开目录。
- 纯预览页菜单中不应出现删除动作。
- 编辑页点击删除后，现有删除流程与多引用流程保持不变。
- 菜单组件接收 action item 模型时，点击结果能回传统一 action key。
- 上下文解析失败时，菜单不应错误弹出。
- 纯预览页在菜单弹出后如果文档上下文已切换，旧菜单动作不能继续按新文档上下文执行。
- 编辑页与纯预览页都需要覆盖菜单关闭行为，确保点击空白、再次右键其他位置、按下 `Esc`、滚动或窗口尺寸变化时，关闭语义保持当前一致。

## 13. 后续扩展方向

本次不实现，但架构需要为后续扩展留好入口：

- 标题右键
- 代码块右键
- Mermaid 右键
- 文本选区右键
- 复制类动作

这些后续能力都应接入同一套“上下文解析 + 动作定义 + profile 裁剪 + 菜单展示”链路，而不是重新创建新的专用菜单组件。
