# 设置页配置草稿防脏化修复设计

## 背景

设置页维护了一份独立于 `store.config` 的本地配置草稿 `config.value`。当前“选择目录”取消时，主进程会返回 `undefined`，设置页却直接把这个值写回 `imgAbsolutePath` / `fileAbsolutePath`，随后深度监听立即提交整份配置，导致 schema 校验失败。

失败后页面只弹提示，不会把本地草稿回滚到真实配置，造成：

1. 用户只是点“取消”也会触发“配置无效”提示。
2. 设置页本地草稿与真实生效配置分叉。
3. 后续任意配置修改都会继续带着这份本地脏状态提交，持续失败。

## 目标

1. 目录选择取消时保持原值，不触发本地草稿污染。
2. 任意配置提交失败时，把设置页本地草稿回滚到 `store.config` 的真实值。
3. 不新增用户可见文案，不修改 schema 和主进程目录选择返回值。
4. 顺手补齐同类风险的最小防御，避免未来再次把非字符串目录选择结果直接写入本地草稿。

## 方案

### 1. 抽取设置页草稿工具

新增一个纯函数工具模块，负责：

- 克隆配置草稿，避免直接共享 store 引用。
- 仅当目录选择结果是字符串时，才写回本地草稿字段。
- 当配置提交失败时，标记下一次同步忽略，并把本地草稿回滚到 `store.config`。

这样可以用现有 `node:test` 做 TDD，而不用额外引入 Vue 组件测试基础设施。

### 2. 设置页接线

设置页中：

- `openDirSelect()` / `openFileDirSelect()` 改为调用“仅接受字符串结果”的辅助函数。
- 深度监听提交配置失败时，不只弹 warning，还要把本地草稿恢复为 `store.config` 的真实值。

### 3. 同类问题边界

本次排查范围内，设置页里“异步选择结果直接写回配置”的入口只有：

- `imgAbsolutePath`
- `fileAbsolutePath`

更广义的同类问题是“本地草稿提交失败后不回滚”，本次会一起收口。

## 验证

1. 目录选择取消时，本地草稿字段保持原值。
2. 配置提交失败时，本地草稿回滚到 store 真值，并且下一次同步不会重复提交这次回滚。
3. 成功选择目录时，仍能正常更新草稿并提交。
