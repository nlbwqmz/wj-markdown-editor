# Electron 端配置层分层重构设计

## 1. 背景

当前 Electron 端配置管理主要集中在 `wj-markdown-editor-electron/src/data/configUtil.js` 中，单文件同时承担了以下职责：

- 计算配置文件路径
- 读取和写入 `config.json`
- 合并默认配置
- 修复旧配置与脏配置
- 维护内存态配置
- 触发配置更新广播

这种实现可以工作，但随着配置项、兼容逻辑和调用方逐步增加，单文件职责已经偏重，后续继续扩展会带来以下问题：

- 配置合法性校验与业务兼容修复耦合，难以独立测试
- IO 逻辑与业务规则耦合，后续替换存储实现成本高
- 更新成功、更新失败与广播时机没有被显式建模
- 用户可见错误提示与主进程错误原因尚未完全分层

本次目标不是更换配置存储库，而是在保持现有外部行为不变的前提下，将配置管理重构为长期可维护的分层结构。

## 2. 目标与非目标

### 2.1 目标

- 保留现有配置文件路径：`Documents/wj-markdown-editor/config.json`
- 保留现有 IPC 契约与调用方式
- 保留旧配置自动兼容能力
- 严格将改动范围限制在 Electron 配置模块，不改变其他功能的可观察行为
- 将配置层拆分为清晰的职责模块
- 为配置读写补上原子写与启动容错
- 将用户可见错误统一收口为 `messageKey`，由前端进行国际化展示
- 为后续扩展 `recent` 或其他持久化模块预留一致的分层模式

### 2.2 非目标

- 本次不切换到 `electron-store`、`conf` 或其他第三方配置库
- 本次不修改 `recent.json` 的实现
- 本次不修改 renderer 到主进程的事件名
- 本次不重构设置页深度监听机制
- 本次不调整配置文件的外部存储位置

## 3. 约束

- 后续实现必须在新分支 `refactor/electron-config-layer` 中进行
- 读取文件默认使用 UTF-8，写入文件必须使用 UTF-8 无 BOM
- 所有用户可见提示文案必须通过前端 i18n 管理，主进程不得直接返回硬编码中文提示
- 必须兼容既有 `config.json` 文件，不能要求用户手动迁移
- 除配置模块本身外，文档会话、recent、资源处理、外部文件变更、导出、截图等其他能力不得发生行为回归

## 4. 现状问题

### 4.1 单文件职责过载

`configUtil.js` 既负责持久化，又负责业务修复和外部广播，导致代码边界不清晰。

### 4.2 校验与修复混杂

当前 `validateConfig()` 同时承担以下责任：

- 合并默认值
- 删除未知字段
- 修复快捷键列表
- 修复旧主题别名

这会导致“合法性校验”和“兼容性纠偏”难以拆分验证。

### 4.3 更新成功语义不够明确

现有更新路径是：

1. 更新内存态
2. 写文件
3. 广播更新

如果后续需要增强失败处理，很容易出现“界面已生效但文件未写成功”的不一致风险。

### 4.4 错误提示层级不清晰

主进程与前端之间缺少统一的配置错误结构，未来若要提示用户“写入失败”“重置失败”等问题，容易把文案硬编码到 Electron 侧。

## 5. 设计概览

本次采用“分层重构，不改变外部契约”的方案。整体分为五层：

1. 默认值层：只定义默认配置
2. Schema 层：只定义结构合法性
3. Repair 层：只负责兼容和纠偏
4. Repository 层：只负责磁盘读写
5. Service 层：只负责初始化、更新、内存态和广播

兼容入口 `src/data/configUtil.js` 保留，对外继续提供现有方法名，内部转调新的 `configService`。

## 6. 模块结构

建议新增目录：

```text
wj-markdown-editor-electron/src/data/
├── defaultConfig.js
├── configUtil.js
└── config/
    ├── configConstants.js
    ├── configSchema.js
    ├── configRepairUtil.js
    ├── configRepository.js
    ├── configSnapshotUtil.js
    └── configService.js
```

### 6.1 `defaultConfig.js`

继续只维护默认配置，不引入 IO、修复、广播逻辑。

### 6.2 `configConstants.js`

统一定义：

- 配置文件目录
- 配置文件路径
- 配置版本号
- 默认文件名

### 6.3 `configSchema.js`

定义配置结构合法性，例如：

- `language` 枚举
- `theme.global` 枚举
- `recentMax` 的数值范围
- `autoSave` 的可选值

Schema 只负责“是否合法”，不负责“怎么修”。

### 6.4 `configRepairUtil.js`

负责把旧配置、脏配置、半合法配置修成当前系统可接受的标准配置，至少包含：

- 合并默认值
- 裁剪未知字段
- 修复 `shortcutKeyList`
- 修复 `theme.preview = github-light`
- 规范化数值与枚举边界
- 填充配置版本号

### 6.5 `configRepository.js`

负责：

- 确保目录存在
- 读取文件
- 原子写入文件
- 处理配置损坏备份

不承担任何业务级修复。

### 6.6 `configSnapshotUtil.js`

统一处理：

- 深拷贝配置
- 配置对象对比
- 返回对外只读快照

避免在各处重复编写 `JSON.parse(JSON.stringify(...))`。

### 6.7 `configService.js`

作为新的主入口，负责：

- 初始化配置
- 获取当前配置
- 更新完整配置
- 更新局部字段
- 持久化成功后再广播
- 输出结构化错误

## 7. 初始化流程

启动时配置初始化流程如下：

1. `ensureConfigDir()`
2. 读取 `config.json`
3. 若文件不存在，则基于默认配置初始化
4. 若 JSON 解析失败，则备份损坏文件并回退默认配置
5. 将读取结果交给 `repair + schema validate`
6. 若修复结果与磁盘内容不一致，则回写规范化配置
7. 将最终配置写入内存态

初始化阶段允许“读失败降级”，目标是保证应用可启动。

## 8. 更新流程

运行期更新流程统一为：

1. 基于当前内存配置生成 `nextConfig`
2. 对 `nextConfig` 执行 `repair + validate`
3. 先写磁盘
4. 写成功后更新内存态
5. 写成功后触发配置变更广播

运行期不允许“写失败也假装成功”。如果写盘失败：

- 内存态保持旧值
- 不触发配置更新广播
- 向调用方返回结构化失败结果

## 9. 原子写策略

配置文件写入采用原子写方案：

- 先在同目录生成临时文件
- 将完整内容写入临时文件
- 写入成功后使用重命名替换目标文件

此策略适用于 Windows 和 Linux 的本地磁盘场景，可显著降低写入中断时损坏配置文件的风险。

补充约束：

- 临时文件必须与目标文件位于同一目录
- 写入编码固定为 UTF-8 无 BOM
- Windows 下应对 `EPERM`、`EBUSY` 等瞬时错误做有限重试

## 10. 错误模型与国际化

### 10.1 主进程错误输出

配置层内部统一使用结构化错误信息，例如：

```js
{
  ok: false,
  reason: 'config-write-failed',
  messageKey: 'message.configWriteFailed',
}
```

其中：

- `reason` 供主进程和测试断言使用
- `messageKey` 供前端国际化展示使用

### 10.2 前端展示原则

只有“用户需要感知或需要决策”的错误才进入前端提示，例如：

- 配置写入失败
- 重置默认配置失败
- 配置目录不可写

以下情况默认只记录日志，不提示用户：

- 启动时自动修复旧字段
- 启动时补齐默认值
- 快捷键排序自动修正
- 旧主题名自动兼容

### 10.3 i18n 落点

需要新增或补齐的国际化 key 建议放在 Web 端 `message` 节点，例如：

- `message.configReadFailed`
- `message.configWriteFailed`
- `message.configResetFailed`
- `message.configDirectoryUnavailable`

Electron 主进程不得直接返回中文提示文本。

## 11. 对现有调用方的影响

### 11.1 `src/main.js`

初始化入口继续调用 `configUtil.initConfig(callback)`，无需改变现有调用方式。

### 11.2 `src/util/channel/ipcMainUtil.js`

继续通过 `configUtil.getConfig()`、`setConfig()`、`setThemeGlobal()`、`setLanguage()` 提供能力。

如需增加失败提示，只在返回结构上增强，不改事件名。

### 11.3 Renderer 设置页

设置页继续使用当前配置回传方式，前端只在收到失败结果时，根据 `messageKey` 执行国际化提示。

### 11.4 对其他功能的影响边界

本次重构只允许影响以下范围：

- 配置初始化
- 配置读写
- 配置修复
- 配置错误提示链路

本次重构不得改变以下功能的既有行为：

- 文档打开、编辑、保存、另存为
- recent 列表维护与打开链路
- 外部文件变更监听与处理
- 资源打开、删除和路径解析
- 导出、截图、图床上传等桌面能力
- 设置页除配置读写失败提示外的交互逻辑

## 12. 分阶段实施建议

### 阶段一：内核模块落地

- 新建 `config/` 目录与基础模块
- 在不改外部 API 的前提下迁移当前逻辑
- 保证初始化和读写行为与现状一致

### 阶段二：补齐原子写与错误模型

- 将写盘统一收口到 `configRepository`
- 引入结构化错误返回
- 明确成功与失败时的内存态切换语义

### 阶段三：补齐国际化与前端提示

- 增加新的 `messageKey`
- 在前端设置页和相关调用点展示失败提示
- 保持可静默修复逻辑不弹窗

### 阶段四：测试补强

- 为 `configRepairUtil` 增加兼容修复测试
- 为 `configRepository` 增加原子写与损坏备份测试
- 为 `configService` 增加初始化、成功更新、失败回滚测试

## 13. 测试策略

至少覆盖以下场景：

- 无配置文件时使用默认配置启动
- 旧配置文件读入后自动修复并规范化回写
- `shortcutKeyList` 缺项、脏项和顺序异常会被修正
- `theme.preview = github-light` 自动修成 `github`
- 写盘失败时内存态不前移
- 写盘失败时不触发更新广播
- 需要前端展示的失败结果带稳定 `messageKey`
- 回归验证配置模块之外的主链路不受影响

建议补充一组最小回归检查，确保本次改动不会波及其他功能：

- 应用启动后仍可正常打开设置页并读取现有配置
- 修改配置后设置广播仍可同步到现有窗口
- 编辑器打开 Markdown、保存、另存为链路不受影响
- recent 记录数量调整外，recent 列表的读写与广播行为保持不变
- 导出、截图、资源操作等依赖 `configUtil.getConfig()` 的调用方仍能读取完整配置快照

## 14. 本次明确不做的事项

- 不迁移到第三方配置库
- 不合并 `recent.json`
- 不修改配置文件路径
- 不将所有内部修复都转成用户可见提示
- 不在本次重构中调整设置页交互模型

## 15. 结论

本次采用“分层重构”而不是“换库重写”的方案，核心原因是：

- 当前项目的真实复杂度在兼容修复和外部契约稳定性，不在 JSON 持久化本身
- 现有外部接口已经较为稳定，适合通过内部换骨架提升可维护性
- 该方案兼顾兼容性、可测试性和后续扩展空间，风险低于直接迁移到新库

后续若配置层重构稳定，再评估是否把 `recent` 等其他持久化模块逐步纳入同样的 `repository + service` 结构。
