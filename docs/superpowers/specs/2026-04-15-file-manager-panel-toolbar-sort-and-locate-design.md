# 文件管理栏工具区合并、排序与当前文档定位增强设计

## 1. 背景

当前文件管理栏已经具备以下基础能力：

- 根据当前文档会话自动解析目标目录并展示当前目录列表
- 支持打开上一级目录
- 支持定位当前文档所在目录
- 支持选择目录
- 支持新建文件夹
- 支持新建 Markdown
- 支持搜索当前目录条目

现有实现主要分布在以下位置：

- Web 组件：[wj-markdown-editor-web/src/components/layout/FileManagerPanel.vue](../../../../wj-markdown-editor-web/src/components/layout/FileManagerPanel.vue)
- Renderer 控制器：[wj-markdown-editor-web/src/util/file-manager/fileManagerPanelController.js](../../../../wj-markdown-editor-web/src/util/file-manager/fileManagerPanelController.js)
- Electron 目录服务：[wj-markdown-editor-electron/src/util/document-session/documentFileManagerService.js](../../../../wj-markdown-editor-electron/src/util/document-session/documentFileManagerService.js)

当前存在 3 个明确改进点：

1. 工具区中“新建文件夹”和“新建 Markdown”占用两个独立按钮，信息密度偏高。
2. 目录列表目前只有固定排序逻辑，无法按名称、修改时间、类型切换升降序。
3. “定位当前文档目录”按钮在当前已处于目标目录时会禁用，无法继续帮助用户把当前文件快速滚动到可视区中部。

## 2. 目标与非目标

### 2.1 目标

- 将两个新建动作合并为一个统一入口，降低工具区按钮数量。
- 为文件管理栏提供可持久化的排序能力，支持名称、修改时间、类型三类排序规则的升降序。
- 将“定位当前文档目录”增强为“定位当前文档文件”，在必要时完成目录跳转、搜索清空与平滑滚动。
- 保持现有文件管理栏目录监听、目录切换和创建流程的主线不变。

### 2.2 非目标

- 本次不新增文件删除、重命名、复制路径、右键菜单等文件管理能力。
- 本次不新增树形目录展开、多选、批量操作。
- 本次不新增“文件夹始终在前”的用户可配置开关，该行为保持为固定规则。
- 本次不引入大小、创建时间、最近打开等额外排序维度。

## 3. 用户确认后的产品决策

### 3.1 新建入口合并

- 顶部工具区中的“新建文件夹”和“新建 Markdown”合并为一个下拉按钮。
- 新按钮图标固定为 `i-tabler:square-plus`。
- 下拉菜单仅包含两项，均不显示图标：
  - 新建文件夹
  - 新建 Markdown
- 菜单项点击后继续复用当前已存在的 `createFolder` 与 `createMarkdown` 创建链路，不改变输入弹窗、校验逻辑与创建副作用。

### 3.2 排序功能

- 顶部工具区新增一个排序下拉按钮。
- 排序按钮图标固定为 `i-tabler:arrows-sort`。
- 排序下拉菜单直接展示完整规则，而不是拆分为“排序字段 + 升降序”两个入口。
- 首版固定支持以下 6 个选项：
  - 名称升序
  - 名称降序
  - 修改时间升序
  - 修改时间降序
  - 类型升序
  - 类型降序
- 排序选择写入全局配置，应用重启后仍保持。
- “文件夹始终在前”作为固定规则保留，不在 UI 中提供单独开关。

### 3.3 当前文档定位增强

- 原“定位当前文档目录”按钮升级为“定位当前文档文件”。
- 当当前存在真实文档路径时，该按钮始终可点击；不再因为“当前已处于文档所在目录”而禁用。
- 行为规则：
  - 如果当前不在文档所在目录：先切换到目标目录，再滚动到目标文件条目。
  - 如果当前已经在文档所在目录：直接滚动到目标文件条目。
  - 如果搜索条件把目标文件过滤掉：先自动清空搜索，再滚动到目标文件条目。
- 滚动要求：
  - 使用平滑滚动。
  - 尽量让目标文件停留在可滚动区域中部。

## 4. 交互设计

### 4.1 工具区布局调整

调整后的文件管理栏工具区按钮顺序建议保持为：

1. 打开上一级目录
2. 定位当前文档文件
3. 选择目录
4. 新建下拉按钮
5. 排序下拉按钮

这样排列的原因是：

- 前 3 个动作属于“目录导航”。
- 新建和排序属于“当前目录操作”。
- 将相近动作聚合到右侧，能减少顶部工具区视觉噪声。

### 4.2 新建下拉按钮交互

- 无目录时，新建下拉按钮保持禁用态。
- 有目录时，点击按钮展开下拉菜单。
- 选择“新建文件夹”后，继续沿用当前文件夹创建弹窗。
- 选择“新建 Markdown”后，继续沿用当前 Markdown 创建弹窗，固定补齐 `.md` 后缀的既有规则不变。

### 4.3 排序下拉按钮交互

- 无目录时，排序按钮仍保持可点击，用于提前切换默认排序偏好，避免工具区布局抖动，也避免用户必须等目录加载后再设置排序。
- 菜单中使用选中态标明当前生效规则。
- 切换排序规则后，当前列表立即按新规则重排，不要求用户重新打开目录或重启窗口。

### 4.4 当前文档定位按钮交互

- 草稿态、无真实磁盘文件路径等无法确定目标文件时，按钮禁用。
- 点击后分两种情况：
  - 当前目录已命中目标目录：直接定位当前文件条目。
  - 当前目录未命中目标目录：切换目录后定位当前文件条目。
- 若当前存在搜索关键字且目标文件被过滤掉：
  - 自动清空搜索输入框
  - 等待过滤结果恢复
  - 再执行定位滚动

## 5. 排序模型设计

### 5.1 固定前置规则

所有排序规则都先执行以下固定前置分组：

1. 目录条目在前
2. 文件条目在后

本规则不暴露为配置开关，属于文件管理栏的固定语义。这样可以保持“先找目录、再找文件”的侧栏导航心智，不把当前面板变成系统资源管理器式的完全混排列表。

### 5.2 排序字段定义

本次支持 3 个排序字段：

- `name`：名称
- `modifiedTime`：修改时间
- `type`：类型

每个字段支持 2 个方向：

- `asc`：升序
- `desc`：降序

### 5.3 二级排序兜底

为保证排序稳定，所有规则在主排序键相同的情况下，都统一回落到名称排序。

建议规则如下：

- 名称升序：目录组按名称升序，文件组按名称升序
- 名称降序：目录组按名称降序，文件组按名称降序
- 修改时间升序：目录组按修改时间升序，文件组按修改时间升序；相同时间再按名称升序
- 修改时间降序：目录组按修改时间降序，文件组按修改时间降序；相同时间再按名称升序
- 类型升序：文件组按业务类型顺序升序，相同类型再按名称升序
- 类型降序：文件组按业务类型顺序降序，相同类型再按名称升序

### 5.4 类型排序定义

类型排序不直接使用原始扩展名字符串，而是复用文件管理栏的业务类型定义。

建议类型顺序为：

1. `directory`
2. `markdown`
3. `image`
4. `video`
5. `pdf`
6. `word`
7. `sheet`
8. `archive`
9. `audio`
10. `other`

这样做可以保证“图标分类”和“排序分类”使用同一套类型语义，避免出现显示类型与排序类型不一致的问题。

## 6. 配置设计

### 6.1 配置结构

在全局配置根对象中新增：

```js
fileManagerSort: {
  field: 'type',
  direction: 'asc',
}
```

字段约束：

- `field` 允许值：`name | modifiedTime | type`
- `direction` 允许值：`asc | desc`

### 6.2 默认值

默认值采用：

```js
fileManagerSort: {
  field: 'type',
  direction: 'asc',
}
```

这样做的原因是当前文件管理栏本就更接近“目录优先 + Markdown 优先 + 其他文件后置”的分组式排序。以 `type + asc` 作为默认值，和当前默认展示顺序最接近，迁移成本最低。

### 6.3 配置接入范围

该字段需要同时接入：

- `defaultConfig`
- `configSchema`
- `configRepairUtil`
- 配置相关测试

原因是本项目的配置更新采用“完整 schema 校验 + repair 回填”的严格模型，新增字段必须同时进入默认值、校验和修复三条链路，才能兼容老配置。

## 7. 数据与模块设计

### 7.1 Electron 目录状态扩展

为了支持修改时间排序，Electron 侧目录项需要补充修改时间元数据。

当前目录项结构建议从：

```js
{
  path,
  name,
  kind,
  extension,
}
```

扩展为：

```js
{
  path,
  name,
  kind,
  extension,
  modifiedTimeMs,
}
```

其中：

- `modifiedTimeMs` 为毫秒时间戳
- 不返回格式化字符串，避免展示逻辑泄漏到主进程

### 7.2 Renderer 排序落点

最终展示排序继续放在 Renderer 侧完成，不新增“切换排序时重新请求主进程排序结果”的协议。

理由：

- 当前文件管理栏最终展示顺序已经由 Renderer 控制器统一整理。
- 排序规则切换属于纯展示层逻辑。
- 将排序放在 Renderer 侧，可以自然覆盖初次加载、切目录、目录监听更新和配置热更新等场景。

### 7.3 类型解析收口

当前 `FileManagerPanel.vue` 中已经定义了图标扩展名分类规则，但这些规则尚未抽成共享能力。

本次建议新增或抽取共享 util，统一负责：

- 解析扩展名
- 解析业务类型
- 解析类型顺序权重
- 解析图标 profile

这样可以保证以下两件事永远一致：

1. 列表图标展示
2. 类型排序结果

## 8. 当前文档定位实现编排

### 8.1 基本流程

点击“定位当前文档文件”按钮后，按以下步骤执行：

1. 解析当前文档真实路径与目标目录
2. 若无真实路径，则直接返回，不触发动作
3. 若当前搜索条件可能隐藏目标文件，则清空搜索
4. 若当前目录不是目标目录，则先切换目录
5. 等待响应式列表与 DOM 渲染完成
6. 将目标文件条目平滑滚动到可视区中部

### 8.2 滚动实现

滚动建议使用浏览器原生 `scrollIntoView`：

```js
element.scrollIntoView({
  behavior: 'smooth',
  block: 'center',
  inline: 'nearest',
})
```

语义依据如下：

- `behavior: 'smooth'`：平滑滚动
- `block: 'center'`：尽量使元素垂直停留在滚动容器中部
- `inline: 'nearest'`：避免不必要的横向滚动

该 API 行为可直接参考 MDN 文档。

### 8.3 DOM 定位方式

Renderer 侧需要为每个条目维护可定位的 DOM 引用，用于“切目录后滚动到当前文件”。

本次固定采用以下实现：

- 为 `v-for` 条目维护 `entry.path -> element` 的映射

原因是：

- 定位动作依赖真实路径，直接映射最稳定
- 不依赖二次查询 DOM，避免选择器与渲染结构耦合过深

### 8.4 异步收口

目录切换与滚动之间不能写成同步直连逻辑，需要等待：

- 目录请求完成
- 列表计算属性更新
- DOM 更新完成

否则会出现“目录已切换，但目标条目节点尚未渲染”的竞态问题。

### 8.5 失败处理

建议的失败处理如下：

- 无真实文档路径：按钮禁用
- 切目录失败：保持当前目录与滚动位置，不额外弹窗
- 切目录成功但目标文件条目不存在：静默收口，不抛异常

## 10. 风险与兼容性

### 10.1 风险

- 若类型解析逻辑继续分散在组件与控制器中，图标与排序会出现不一致。
- 若“切目录后滚动”未等待 DOM 更新，定位会间歇性失效。
- 若配置新增字段未接入 repair 链路，老配置升级时可能被 schema 拒绝。

### 10.2 兼容性策略

- 保持创建动作、目录切换、目录监听等既有主线不变。
- 排序作为显示层增强，不改变 `openDirectory`、`createFolder`、`createMarkdown` 的 IPC 契约。
- 仅为目录项追加 `modifiedTimeMs` 元数据，不破坏现有字段。

## 11. 实施顺序建议

建议按以下顺序实施：

1. 抽取文件管理栏条目类型解析与排序 util
2. 扩展 Electron 目录项返回 `modifiedTimeMs`
3. 接入 `fileManagerSort` 默认配置、schema 与 repair
4. 改造 `FileManagerPanel.vue` 顶部工具区为“新建下拉 + 排序下拉”
5. 改造 Renderer 侧目录排序逻辑，按配置动态排序
6. 增强“定位当前文档文件”按钮的滚动编排
7. 补齐 Web / Electron / 配置测试

## 12. 结论

本次设计在不扩展文件管理栏职责边界的前提下，聚焦三项高频操作增强：

- 合并新建入口，降低工具区复杂度
- 引入可持久化的排序能力，增强目录浏览效率
- 将“定位当前文档目录”升级为“定位当前文档文件”，强化当前文档回到可视区的效率

整体方案遵循当前项目的既有架构：

- 目录数据仍由 Electron 提供
- 展示排序仍由 Renderer 收口
- 配置仍经统一配置链路持久化
- 创建与目录切换仍复用既有文件管理栏主线
