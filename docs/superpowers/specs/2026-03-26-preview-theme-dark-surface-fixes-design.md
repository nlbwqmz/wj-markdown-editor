# 预览主题暗黑表面问题修复设计

## 1. 背景

本次需求来自用户对预览主题真实使用效果的反馈，问题集中在少数主题的表格、引用块和暗黑模式媒体控件表现上。现有预览主题体系已经收口到“变量协议 + 基础骨架 + 主题特例”的结构，因此本轮不应通过新增配置项或修改运行时逻辑来修复，而应直接在主题样式层完成定点治理。

用户明确指出的问题如下：

- `juejin` 主题明亮模式下表格样式过丑
- `juejin` 主题暗黑模式下引用块左侧竖条不清晰
- `juejin` 主题暗黑模式下表格样式过丑
- `scrolls` 主题暗黑模式下表格样式过丑
- 除 `github` 外，其他预览主题在暗黑模式下的音频控件缺少优化，希望统一接近 `github` 暗黑模式下的观感

这些问题都属于样式语义与局部主题覆写不协调，不涉及 Markdown 渲染链路、配置项、主题切换逻辑或 Electron 侧会话流程。

## 2. 目标

本轮修复目标如下：

1. 仅通过 Web 端预览主题样式修复用户点名的问题。
2. 保持 `juejin` 与 `scrolls` 主题原有视觉人格，不将它们直接改造成 `github` 表格风格。
3. 统一非 `github` 主题在暗黑模式下的音频控件观感，使其接近 `github` 主题在暗黑模式下的原生控件表现。
4. 不新增设置项，不调整主题枚举，不改动主题切换逻辑，不引入新的运行时分支。
5. 为本轮问题补充最小必要的自动化约束，防止后续同类回退。

## 3. 非目标

本轮明确不做以下事项：

- 不重构预览主题变量协议。
- 不新增预览主题，也不修改预览主题入口顺序。
- 不修改 Markdown 渲染器、资源协议或 Electron 主进程逻辑。
- 不为每个主题单独定制一套暗黑音频控件皮肤。
- 不扩大到其他未被用户点名的视觉问题。

## 4. 根因分析

### 4.1 `juejin` 表格问题

`juejin` 主题仍保留了较早期的表格表现组合：

- `12px` 字号过小
- `inline-block` + `auto` 宽度让整体更像临时容器，而不是现代正文表格
- 单元格边框完全移除，导致结构感不足
- 表头、正文和奇偶行层次过弱

这组配置在明亮模式下会显得“轻但散”，在暗黑模式下则进一步放大层次不足的问题。

### 4.2 `juejin` 暗黑引用块问题

`juejin` 暗黑模式中的引用块边条直接跟随通用边框语义，颜色对比不足。正文背景和边条缺乏明确层次，导致引用块在暗黑模式下难以快速识别。

### 4.3 `scrolls` 暗黑表格问题

`scrolls` 主题的表格本身保留了卷轴风格，但暗黑分支只做了少量变量覆盖：

- 边框清晰度不够
- 表头背景存在但不够稳定
- 行层次偏弱
- `12px` 正文与较窄 padding 在暗黑模式下会显得发挤

因此问题不是主题方向错误，而是暗黑分支表格表面信息不足。

### 4.4 非 `github` 主题暗黑音频控件问题

当前其他主题没有统一的暗黑媒体控件约束，原生 `audio` 控件只能被动继承宿主环境。`github` 主题在暗黑模式下由于 `color-scheme` 和周边表面语义更完整，因此看起来更协调。根因在于“非 `github` 主题缺少统一的暗黑原生媒体控件约束”，而不是音频 Markdown 语法本身有问题。

## 5. 设计结论

采用“方案 A：定点修复”，只改主题样式，不碰逻辑层。

设计原则如下：

- 优先复用现有变量合同与稳定根选择器。
- 能通过主题变量表达的内容，尽量不额外添加元素级硬编码。
- 只有当变量不足以完成目标表面时，才在主题文件中增加少量局部选择器覆写。
- 音频控件统一采用“共享暗黑媒体约束”的方式处理，不为每个主题单独复制一套 `github` 规则。

## 6. 方案设计

### 6.1 `juejin` 表格修复

`juejin` 表格修复遵循“保留简洁、增强结构”的方向：

- 提高表格正文可读性，不再维持当前偏小字号。
- 强化表头与正文层次，保留低饱和灰白基调。
- 为单元格恢复适度边框或等价结构感，不再完全依赖纯留白分隔。
- 保留斑马纹，但让奇偶行层次更加稳定，避免明亮模式与暗黑模式都显得松散。
- 表格容器继续支持横向滚动，但不再表现成明显“老旧卡片化”容器。

这部分主要落在 `juejin.scss` 的表格变量与少量局部规则上。

### 6.2 `juejin` 暗黑引用块修复

引用块只做最小必要增强：

- 提高左侧边条与背景之间的对比度。
- 保持正文文字色仍为克制的次级文本，不把引用块改成强提示框。
- 背景层次只做轻度增强，避免喧宾夺主。

这部分仅调整 `juejin` 暗黑分支中的 `blockquote` 相关变量，不扩散到基础骨架。

### 6.3 `scrolls` 暗黑表格修复

`scrolls` 主题表格保持卷轴暖色方向，但修正暗黑模式下的信息密度和层次：

- 增强边框清晰度，使表格结构不再发糊。
- 调整表头背景与正文对比，保持暖色但避免泛黄发脏。
- 增强奇偶行层次，让表格在暗黑模式下更容易扫描。
- 适度放宽字号和单元格节奏，降低“发挤”的感受。

这部分主要修改 `scrolls.scss` 的暗黑表格变量与必要的局部规则，保留卷轴底纹和标题装饰。

### 6.4 非 `github` 主题暗黑音频控件统一

音频控件采用共享暗黑约束，而不是主题逐一重写：

- 命中范围限定为“暗黑模式 + 非 `github` 预览主题 + `audio` 元素”。
- 通过 `color-scheme` 与周边表面约束，让原生媒体控件尽量接近 `github` 暗黑主题下的观感。
- 不影响亮色模式，不影响 `github` 主题，不改变 Markdown 渲染结构。

实现上优先放在预览主题基础层或合同层中，以稳定根选择器表达；若基础层无法无歧义表达“排除 `github`”范围，则允许在入口附近增加一段共享样式，但必须保持稳定根和最小命中范围。

## 7. 影响文件

预计涉及以下文件：

- `wj-markdown-editor-web/src/assets/style/preview-theme/theme/juejin.scss`
- `wj-markdown-editor-web/src/assets/style/preview-theme/theme/scrolls.scss`
- `wj-markdown-editor-web/src/assets/style/preview-theme/preview-theme-base.scss` 或 `wj-markdown-editor-web/src/assets/style/preview-theme/preview-theme-contract.scss`
- `wj-markdown-editor-web/src/assets/style/__tests__/previewThemeStructure.test.js`

是否修改基础层，取决于共享暗黑音频约束的落点是否能保持结构简洁；若无法在不污染基础语义的前提下表达，则只允许新增一小段共享选择器，不新增新主题文件。

## 8. 验证设计

### 8.1 自动化验证

本轮自动化验证只补最小必要保护：

- 增加或扩展预览主题结构测试，约束新增的暗黑音频规则只能命中“非 `github` 主题 + 暗黑模式 + `audio`”。
- 保持已有预览主题稳定根、入口顺序和变量协议测试不被破坏。

不新增视觉截图测试，不引入浏览器端 E2E。

### 8.2 手工验收点

至少检查以下组合：

- `juejin` 明亮模式表格
- `juejin` 暗黑模式表格
- `juejin` 暗黑模式引用块
- `scrolls` 暗黑模式表格
- `github` 暗黑模式音频控件，作为对照样本
- 任意非 `github` 主题暗黑模式音频控件，确认已接近对照样本

重点关注：

- 表格是否更规整、更容易阅读
- `scrolls` 是否仍保留卷轴主题气质
- 引用块边条是否足够清晰但不过度抢眼
- 音频控件是否仅在暗黑模式下被修正，且未误伤 `github`

## 9. 风险与控制

### 9.1 风险：主题人格被削弱

控制方式：

- `juejin` 保持克制的灰白语义，只增强结构。
- `scrolls` 保持暖色与卷轴装饰，不直接改成 `github` 表格。

### 9.2 风险：暗黑音频控件选择器命中过宽

控制方式：

- 自动化测试显式校验命中范围。
- 规则中显式排除 `github` 主题，并限定在暗黑模式下。

### 9.3 风险：为音频控件新增规则后污染基础层职责

控制方式：

- 若基础层表达不自然，则用最小共享规则承接，不把它硬塞进无关变量协议。
- 不借本轮顺手扩展更多媒体控件能力。

## 10. 完成标准

满足以下条件即可进入实现：

- 修复范围严格限定在样式层。
- `juejin` 明亮 / 暗黑表格、暗黑引用块的修复策略明确。
- `scrolls` 暗黑表格的修复策略明确。
- 非 `github` 主题暗黑音频控件统一策略明确。
- 自动化验证与手工验收点已定义，且没有把需求扩大到其他未确认问题。
