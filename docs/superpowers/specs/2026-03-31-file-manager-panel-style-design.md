# 文件管理栏样式与导航补充设计

## 1. 背景

主窗口文件管理栏已经落地，但当前交互与视觉上还有几处不统一：

- 文件管理栏列表滚动条未统一使用项目侧栏滚动样式。
- 文件管理栏顶部与外层拖动条顶部缺少与编辑区、预览区一致的上边框。
- 编辑页在“大纲位于左侧”场景下，大纲容器右侧缺少统一边框，导致分区感不一致。
- 文件管理栏当前文件仍使用背景块高亮，和现有轻量侧栏导航风格不一致。
- 文件管理栏只能进入子目录，不能显式返回上一级。
- 新建文件夹的“目标已存在”提示链路在 renderer 侧已经具备，但 Electron `createFolder()` 还没有像 `createMarkdown()` 一样稳定返回 `file-manager-entry-already-exists`。
- 文件管理栏关闭后仍保留箭头唤起区，与本轮期望的“完全收起”不一致。

本次需求只处理上述样式与导航补充，不改动文件管理栏主干架构，不新增新的主进程协议，不重做编辑页分栏系统。

## 2. 目标与非目标

### 2.1 目标

1. 让文件管理栏滚动、边框和当前项高亮方式与主窗口现有视觉规则一致。
2. 为文件管理栏增加明确的“返回上一级”能力，并复用现有目录切换链路。
3. 补齐创建文件夹在“目标已存在”时的主进程失败返回，让现有 renderer 提示链路能稳定生效。
4. 去掉文件管理栏关闭后的左侧箭头唤起区，保持关闭语义简单直接。

### 2.2 非目标

1. 不新增文件树、多级目录面板或目录历史。
2. 不改动主进程 `document-session` / runtime 命令集合。
3. 不新增新的重新打开入口；文件管理栏关闭后的重新打开仍依赖现有菜单或快捷键。
4. 不重做编辑页 split-grid 轨道、拖拽算法或大纲组件结构。

## 3. 用户确认的交互决策

### 3.1 返回上一级入口

用户已确认采用独立按钮方案：

- 在文件管理栏工具栏中新增一个“上一级”按钮。
- 不采用“仅靠面包屑点击返回”的方案。

### 3.2 左侧大纲边框

用户已确认：

- 当编辑页为左侧大纲布局时，大纲容器右侧需要增加边框。
- 边框样式必须与当前系统其他分隔边框保持一致。

## 4. 视觉设计

### 4.1 文件管理栏滚动

- 文件管理栏列表区域改用 `wj-scrollbar`。
- 仅列表滚动区域使用该类；工具栏与外层容器继续保持 `overflow-hidden`。
- 该调整只影响文件管理栏内部滚动观感，不改变主窗口整体滚动约束。

### 4.2 文件管理栏顶部边框

文件管理栏顶部要与编辑区、预览区上沿视觉对齐，因此补齐两处边框：

1. 文件管理栏宿主容器顶部增加统一边框。
2. 文件管理栏外层拖动条顶部增加统一边框。

边框规则：

- 颜色使用现有 `border-primary` 对应变量。
- 线型使用项目现有实线边框风格。
- 不引入新的主题变量。

### 4.3 左侧大纲边框

仅当编辑页处于“左侧大纲”布局时：

- 大纲容器右侧增加统一 `border-right`。
- 颜色、线宽、线型与系统既有边框一致。

右侧大纲布局不新增额外边框，因为当前右侧场景已通过分隔条满足视觉分区。

### 4.4 当前文件高亮

文件管理栏当前文件的高亮从“背景块 + 描边”调整为“文字颜色高亮”：

- 当前文件项不再使用专门的背景色块作为激活态主体表达。
- 当前文件名称和必要时的图标颜色切换到高亮文字色。
- hover 背景仍保留，避免列表失去可点击反馈。

设计目的：

- 降低当前项在侧栏中的视觉重量。
- 让当前项更接近“导航定位”而不是“已选标签页”。

## 5. 交互设计

### 5.1 返回上一级

文件管理栏工具栏新增“上一级”按钮，交互规则如下：

1. 当前没有目录时，按钮禁用。
2. 当前目录已经是根目录时，按钮禁用。
3. 计算得到的父目录与当前目录相同，按钮禁用，避免无效跳转。
4. 点击后复用现有 `openDirectory(targetPath)` 能力切换目录。

实现约束：

- 按钮沿用现有文件管理栏工具按钮风格。
- 补齐对应 i18n 文案。
- 提供稳定的 `data-testid`，方便组件测试直接断言按钮状态与点击行为。

支持的根目录语义沿用现有路径工具逻辑：

- `/`
- `C:/`
- UNC share root，例如 `//server/share`

### 5.2 面包屑

- 本次不把面包屑改为逐级可点击返回。
- 面包屑继续承担当前位置展示职责。

### 5.3 文件管理栏关闭态

- 文件管理栏关闭后，不再渲染左侧箭头唤起区。
- 关闭后的主内容区直接占满原宿主区域。
- 重新打开文件管理栏继续依赖现有 View 菜单或快捷键入口。

这样可以避免关闭态仍残留一个窄条区域，影响主窗口左缘整洁度。

## 6. 实现设计

### 6.1 Renderer 组件落点

#### `wj-markdown-editor-web/src/components/layout/FileManagerPanel.vue`

负责以下调整：

- 工具栏新增“上一级”按钮。
- 列表滚动容器增加 `wj-scrollbar`。
- 当前文件激活态从背景高亮改为文字颜色高亮。
- 文件管理栏顶部样式与新按钮状态展示。

#### `wj-markdown-editor-web/src/util/file-manager/fileManagerPanelController.js`

负责以下逻辑：

- 基于当前 `directoryPath` 计算是否允许返回上一级。
- 直接复用该文件内现有的路径归一化与父目录计算语义，不新增另一套路径规则。
- 暴露 `openParentDirectory()` 或同等语义方法给组件调用。
- 复用已有 `openDirectory()` 执行目录切换。
- 继续复用现有 `file-manager-entry-already-exists` 失败提示链路，不重复新增另一套前端错误处理。

#### `wj-markdown-editor-web/src/views/HomeView.vue`

负责以下外层壳层调整：

- 文件管理栏宿主顶部补边框。
- 文件管理栏外层 gutter 顶部补边框。
- 文件管理栏关闭后移除唤起按钮区域，主内容直接占满。

#### `wj-markdown-editor-web/src/components/editor/MarkdownEdit.vue`

负责以下布局态样式补充：

- 当布局模式为左侧大纲时，为大纲容器增加右侧边框 class。
- 不改动现有 gutter 顺序、列模板和 split 初始化逻辑。

#### `wj-markdown-editor-web/src/components/editor/MarkdownMenu.vue`

作为大纲容器本体，承接统一边框样式，不改变锚点渲染逻辑。

### 6.2 主进程落点

#### `wj-markdown-editor-electron/src/util/document-session/documentFileManagerService.js`

补充以下约束：

- `createFolder()` 在目标已存在时显式返回：
  - `ok: false`
  - `reason: 'file-manager-entry-already-exists'`
  - `path: <目标路径>`

其中“目标已存在”同时包含两类情况：

- 同名目录已存在
- 同名文件已存在

这样 renderer 可以与新建 Markdown 复用同一套“已存在”提示逻辑。

不新增“返回上一级”主进程命令，因为该需求可通过 renderer 计算父目录后调用既有目录打开命令完成。

## 7. 数据流与状态规则

### 7.1 返回上一级链路

1. `FileManagerPanel` 点击“上一级”按钮。
2. `fileManagerPanelController` 基于当前 `directoryPath` 计算父目录。
3. 若父目录不可用则直接返回 `no-op` 风格结果。
4. 若父目录有效，则调用现有 `openDirectory(parentPath)`。
5. 目录切换成功后继续由现有目录状态归一化逻辑刷新面包屑与列表。

### 7.2 新建文件夹已存在链路

1. 用户输入新建文件夹名称。
2. renderer 调用现有 `requestFileManagerCreateFolder()`。
3. Electron 若发现同名目录或同名文件已存在，返回 `file-manager-entry-already-exists`。
4. renderer 复用已有统一提示链路，并保持当前目录状态不变。

### 7.3 文件管理栏关闭链路

- `HomeView` 仍按 `store.fileManagerPanelVisible` 判断是否渲染文件管理栏。
- 当该值为 `false` 时：
  - 不渲染文件管理栏插槽。
  - 不渲染 gutter。
  - 不渲染左侧箭头唤起按钮。
  - 只保留主内容容器。

## 8. 错误处理

### 8.1 返回上一级

以下情况直接禁用按钮，不发请求：

- 当前无目录。
- 当前目录为根目录。
- 父目录解析失败。
- 父目录与当前目录相同。

### 8.2 创建文件夹

以下失败路径需要显式保留当前目录状态：

- `invalid-file-manager-entry-name`
- `file-manager-entry-already-exists`
- `open-directory-watch-failed`

其中：

- `file-manager-entry-already-exists` 由现有 renderer 提示链路统一提示用户目标已存在。
- `open-directory-watch-failed` 沿用现有目录打开失败提示。

### 8.3 关闭态入口

移除箭头唤起区后，不额外补偿新的局部入口；这是有意收紧为“单一重新打开路径”的设计，而不是遗漏。

## 9. 测试设计

### 9.1 Web 端测试

重点补以下回归：

1. 文件管理栏列表滚动容器应包含 `wj-scrollbar`。
2. 当前文件高亮应使用新的激活 class / 文字高亮语义，而不是旧背景高亮语义。
3. 有父目录时“上一级”按钮可点击，点击后调用 `requestFileManagerOpenDirectory` 打开父目录。
4. 根目录或无目录时“上一级”按钮禁用。
5. 继续覆盖 `createFolder()` 收到 `file-manager-entry-already-exists` 时，应提示用户并保持当前目录状态，确认 renderer 复用链路未被破坏。
6. `HomeView` 在文件管理栏关闭时不再渲染左侧唤起按钮。
7. 左侧大纲布局应带统一右边框 class。

### 9.2 Electron 端测试

重点补以下回归：

1. `createFolder()` 命中已存在目录时，必须返回 `file-manager-entry-already-exists`。
2. `createFolder()` 命中同名文件时，也必须返回 `file-manager-entry-already-exists`。
3. 命中已存在目标时，不能覆盖既有文件或目录内容，也不能误返回成功目录状态。

## 10. 实施顺序建议

1. 先补 Electron `createFolder()` 已存在失败测试与实现，覆盖“同名目录 / 同名文件”两种分支。
2. 再补 renderer controller 的“返回上一级”能力，并验证现有“已存在提示”链路继续可用。
3. 调整 `FileManagerPanel.vue` 的工具栏、滚动类和高亮样式。
4. 调整 `HomeView.vue` 的关闭态与顶部边框。
5. 调整 `MarkdownEdit.vue` / `MarkdownMenu.vue` 的左侧大纲右边框。
6. 补齐相关定向测试并做对应包内 ESLint 修复。

## 11. 最终结论

本轮采用“局部补齐式”方案最合适：

- 只修改文件管理栏、大纲容器、主窗口壳层和文件管理服务的局部行为。
- 不引入新的 runtime 命令，不扩展文件管理模型。
- 用最小改动完成滚动统一、边框统一、导航补充和错误反馈补齐。

该方案既满足本轮 7 个明确需求，也能把风险限制在已有测试覆盖较集中的文件管理栏与编辑页布局层。
